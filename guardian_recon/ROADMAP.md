## المرحلة 1 — تثبيت الأساس (Foundation Hardening)
**الهدف:** تحويل اللي بنيناه من "ديمو يشتغل" إلى "نظام إنتاجي يُعتمد عليه"

- [x] 1.1 استبدال التخزين في الذاكرة بقاعدة بيانات حقيقية (SQLite) ✅ **مختبر فعلياً**: تشغيل تسوية → موافقة → إيقاف السيرفر كلياً → تشغيل سيرفر جديد على منفذ مختلف → البيانات والقرارات محفوظة بالكامل
- [ ] 1.2 نظام تسجيل دخول وصلاحيات (Auth) — حتى لو مستخدم واحد بالبداية (أنت)
- [x] 1.3 ربط `OdooConnector` بحساب Odoo حقيقي واختباره ✅ **مختبر فعلياً عبر GitHub Actions**: اتصال ناجح + قراءة فعلية (5 مستخدمين بقاعدة البيانات) — الأسرار بـ GitHub Secrets، لا بيانات حساسة بالكود أو بأي سجل
- [ ] 1.4 اختبار محرك التسوية على كشف حساب بنكي حقيقي كامل (مو بيانات وهمية)
- [x] 1.5 نظام تسجيل أخطاء ومراقبة (logging) احترافي بدل print ✅ مطبّق في المحرك واللوحة
- [x] 1.6 رفع المشروع على GitHub + CI أساسي ✅ https://github.com/MoatasimFaroog1111/Guardianrecon — 12 اختبار آلي + workflow اختبار اتصال Odoo، كلاهما يشتغل بنجاح

**نقطة التحقق:** تقدر تشغّل تسوية حقيقية بالكامل من كشف حساب فعلي → لوحة → موافقة → قيد Odoo فعلي (مسودة)، وكل شي محفوظ بقاعدة بيانات ما يضيع لو انطفى السيرفر.
**الباقي لإكمال المرحلة:** 1.2 (Auth) + 1.4 (يحتاج كشف حساب بنكي فعلي منك — كود الحساب البنكي بأودو + ملف CSV/Excel حقيقي).

## المرحلة 2 — محرك القيود الآلي الكامل (GL Automation)
**الهدف:** أي معاملة (بنكية أو غيرها) تتحول لقيد محاسبي صحيح تلقائياً

- [ ] 2.1 محرك تصنيف الحسابات الذكي (يقرر الحساب المدين/الدائن المناسب) — مبني على `journal-entries-ar` وربطه بـ ML لاحقاً
- [ ] 2.2 التحقق من توازن القيد قبل الإرسال لأودو (Debit = Credit إلزامي)
- [ ] 2.3 قوالب قيود جاهزة للمعاملات المتكررة (رواتب، إيجار، اشتراكات)
- [ ] 2.4 ربط مباشر بمحرك التسوية (Adjustment Required → قيد جاهز تلقائياً)

**نقطة التحقق:** معاملة بنكية جديدة تدخل النظام وتطلع كقيد Odoo متوازن ومصنّف صح بدون تدخل يدوي، وتحتاج بس موافقتك.

## المرحلة 3 — الذمم المدينة والدائنة (AP/AR Automation)
**الهدف:** إدارة كاملة للموردين والعملاء بدون متابعة يدوية

- [ ] 3.1 استخراج الفواتير من PDF/صور (OCR) — نستفيد من الـ pipeline الموجود بمشروعك القديم
- [ ] 3.2 مطابقة الفواتير بأوامر الشراء (3-way matching)
- [ ] 3.3 تنبيهات استحقاق تلقائية (فواتير مستحقة، متأخرة)
- [ ] 3.4 تقرير أعمار الديون (Aging Report) لكل من AP و AR

**نقطة التحقق:** ترفع صورة فاتورة → النظام يستخرجها، يطابقها، يصنفها، ويحطها بلوحتك للموافقة.

## المرحلة 4 — القوائم المالية الآلية (Financial Statements)
**الهدف:** قوائم مالية جاهزة أي لحظة بدون إعداد يدوي

- [ ] 4.1 قائمة الدخل (P&L) تلقائية من بيانات Odoo
- [ ] 4.2 الميزانية العمومية (Balance Sheet) تلقائية
- [ ] 4.3 قائمة التدفقات النقدية (Cash Flow Statement)
- [ ] 4.4 مقارنة الفترات (شهر بشهر، ربع بربع) + تحليل انحرافات تلقائي بالسرد (LLM)
- [ ] 4.5 تصدير تلقائي بصيغة Word/PDF احترافية (نستخدم `python-docx`/`reportlab`)

**نقطة التحقق:** تطلب "قائمة دخل يونيو" وتطلع لك خلال ثواني، جاهزة ومنسقة ومقارنة بالشهر اللي قبله.

## المرحلة 5 — التنبؤ والتحليل المالي (Forecasting & Analytics)
**الهدف:** النظام يشوف المستقبل، مو بس يسجل الماضي

- [ ] 5.1 نموذج تنبؤ تدفقات نقدية (ARIMA/Prophet) على بيانات GL التاريخية
- [ ] 5.2 لوحة KPIs مالية حية (السيولة، دوران المخزون، هامش الربح...)
- [ ] 5.3 تنبيه استباقي لمشاكل السيولة المتوقعة قبل حدوثها
- [ ] 5.4 تحليل موازنة مقابل الفعلي (Budget vs Actual) تلقائي

**نقطة التحقق:** النظام ينبهك "بعد 45 يوم متوقع نقص سيولة" قبل ما يصير فعلاً.

## المرحلة 6 — الامتثال والتدقيق (Compliance & Audit)
**الهدف:** جاهزية دائمة للزكاة/الضريبة والتدقيق بدون هلع آخر السنة

- [ ] 6.1 حساب ضريبة القيمة المضافة تلقائياً من القيود ومطابقتها مع إقرارات هيئة الزكاة
- [ ] 6.2 أثر تدقيق كامل (من غيّر أي شي، متى، ليش) — كل قرار بلوحتك مسجل أصلاً
- [ ] 6.3 تقارير تسوية جاهزة بصيغة تدقيق احترافية (GL-to-Subledger, Bank, Intercompany) حسب منهجية `finance:reconciliation`
- [ ] 6.4 نظام صلاحيات فصل المهام (Segregation of Duties)

**نقطة التحقق:** المدقق الخارجي يطلب أي تقرير تسوية أو أثر تدقيق ويطلع جاهز بضغطة.

## ⚖️ قواعد الالتزام الصارمة (لا استثناءات)

1. **لا نبدأ مرحلة جديدة إلا بعد اجتياز "نقطة التحقق" للمرحلة الحالية فعلياً**
2. **كل كود يُختبر فوراً بعد كتابته** (زي ما سوينا بالتسوية واللوحة) — مو "نظرياً لازم يشتغل"
3. **أي قرار مالي حقيقي يحتاج موافقتك اليدوية** — النظام يقترح، أنت تعتمد، دايماً
4. **لا تعديل مزدوج**: لو عندك نظام مشابه شغال إنتاجياً (Guardian AI Accountant الحالي)، نتكامل معه أو نستبدله بوعي — مو نبني نسخة موازية بالغلط
5. **توثيق مستمر**: كل مرحلة تتحدث بهذا الملف فور اكتمالها (✅ + نسبة % محدثة)
6. **الأولوية دايماً للمرحلة الأقرب رقمياً** — إذا طلبت شي من مرحلة متأخرة، بنّبهك إننا لسا ما خلصنا اللي قبلها
